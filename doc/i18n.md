# Internationalization (i18n) Implementation Guide

This document explains how the internationalization (i18n) system is implemented in the Call-me application.

## Overview

The application now supports multiple languages using a JSON-based translation system. The implementation follows best practices from the Crowdin Node.js i18n guide.

## Supported Languages

Supported locales are discovered dynamically from the JSON files in `app/locales/*.json`.

At the time of writing, the repository ships with:

- **English (en)** - Default
- **Spanish (es)**
- **French (fr)**
- **Italian (it)**
- **German (de)**
- **Portuguese (pt)**
- **Russian (ru)**
- **Arabic (ar)**
- **Hindi (hi)**
- **Chinese (zh)**
- **Japanese (ja)**

## Architecture

### Backend (Server-side)

#### Dependencies

- **i18n**: Node.js internationalization library

#### Configuration

The backend reads locale files from `app/locales/` and configures the `i18n` library based on what JSON files exist.

See: [app/server.js](../app/server.js)

```javascript
// Locales directory location
const LOCALES_DIR = path.join(__dirname, 'locales');

function getAvailableLocales() {
    return fs
        .readdirSync(LOCALES_DIR, { withFileTypes: true })
        .filter((dirent) => dirent.isFile() && dirent.name.endsWith('.json'))
        .map((dirent) => path.basename(dirent.name, '.json'))
        .filter(Boolean)
        .sort();
}

function configureI18n() {
    const locales = getAvailableLocales();
    const effectiveLocales = locales.length > 0 ? locales : ['en'];
    const defaultLocale = effectiveLocales.includes('en') ? 'en' : effectiveLocales[0] || 'en';

    i18n.configure({
        locales: effectiveLocales,
        defaultLocale: defaultLocale,
        directory: LOCALES_DIR,
        objectNotation: true,
        updateFiles: false,
        syncFiles: false,
        api: {
            __: 'translate',
            __n: 'translateN',
        },
    });

    return { locales: effectiveLocales, defaultLocale };
}

configureI18n();
```

#### API Endpoints

**GET** `/locales`

Returns the list of available locales (derived from `app/locales/*.json`):

```json
{
    "locales": ["ar", "de", "en", "es", "fr", "hi", "it", "ja", "pt", "ru", "zh"]
}
```

**GET** `/translations/:locale`

Returns translations for the specified locale:

```json
{
    "locale": "en",
    "translations": {
        "appName": "Call-me",
        "signIn": {
            "title": "Sign In",
            "username": "Enter username"
        }
    }
}
```

### Frontend (Client-side)

#### Core Files

- **[public/i18n.js](../public/i18n.js)**: Client-side i18n library
- **[app/locales/\*.json](../app/locales/)**: Translation files

#### Key Features

1. **Automatic Language Detection**
    - Checks localStorage for saved preference
    - Falls back to browser language
    - Defaults to English if unsupported

    The client fetches supported locales from the server (`GET /locales`). If that request fails, it uses a small fallback list.

2. **Translation Function**

    ```javascript
    t('signIn.username'); // Returns: "Enter username"
    t('room.userJoined', { username: 'John' }); // Returns: "John joined the call"
    ```

3. **HTML Attributes**
    - `data-i18n`: Translates element text content
    - `data-i18n-placeholder`: Translates input placeholder
    - `data-i18n-title`: Translates element title/tooltip

## Adding New Languages

1. Create a new JSON file in `app/locales/` (e.g., `app/locales/nl.json`).
2. Copy the structure from `app/locales/en.json` and translate the values.
3. Restart the server.

Optional:

- If you want a friendly name/flag in the Settings dropdown, add it to the `getLocaleLabel()` mapping in `public/i18n.js`. Otherwise, the dropdown will show the raw locale code.
- If you set `I18N_WATCH=true` in your environment, the server will re-scan `app/locales/` and reconfigure i18n automatically when locale files change.

## Adding New Translation Keys

1. Add the key to all language files in `locales/`:

    ```json
    {
        "newSection": {
            "newKey": "Translation text"
        }
    }
    ```

2. Use in HTML with data attributes:

    ```html
    <button data-i18n="newSection.newKey">Default Text</button>
    ```

3. Or use in JavaScript:
    ```javascript
    const text = t('newSection.newKey');
    ```

## Dynamic Text Replacement

For translations with dynamic values, use placeholder syntax:

**Translation file:**

```json
{
    "room": {
        "userJoined": "__username__ joined the call"
    }
}
```

**Usage:**

```javascript
const message = t('room.userJoined', { username: 'Alice' });
// Result: "Alice joined the call"
```

## Best Practices

1. **Key Naming**: Use dot notation for nested keys (e.g., `settings.language`)
2. **Consistency**: Keep the same structure across all language files
3. **Placeholders**: Use `__placeholder__` format for dynamic values
4. **Fallbacks**: Always provide English translations as fallback
5. **Context**: Group related translations under common parent keys

## Troubleshooting

**Issue**: Translations not loading

- Check browser console for errors
- Verify the locale file exists in `app/locales/` directory
- Ensure the API endpoint is accessible

**Issue**: Some text not translating

- Verify the HTML element has the correct `data-i18n` attribute
- Check that the translation key exists in the locale file
- Ensure `translatePage()` is called after DOM updates

**Issue**: Language not persisting

- Check browser localStorage
- Ensure the locale exists in `app/locales/` and is returned by `GET /locales`
